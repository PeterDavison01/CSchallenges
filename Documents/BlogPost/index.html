This is a guide on how to use the GitHub repository found here. Please follow along if needed.

Since the beginning of the year, I have been working on an Augmented Reality application that allows you to use multiple markers to display Information panels for any real-world Object the user chooses. By physically sticking a marker on an object, the user can use the application to enter details about the IRL object, and then when said marker is scanned using a device's camera, the object's details will come up in an Augmented Reality Information Panel on their device.

The application utilises the Flask web framework library in Python and the lightweight Augmented Reality library AR.js to produce an interface for the user that lets them customise all their objects. This project also uses a Raspberry Pi Model B+ to run the Flask framework, enabling users to connect to the AR app over their local internet.

Navigating the Repository

First of all, you need to find your way around the repository. If you follow the link at the top of the page, you will be brought to the home page of the repo, where you can find 4 items: the README.md, the testerApplication.html, the sourceCode folder and the Documents folder. The README.md contains a description of the overall project and all the files found in the repo. This will be useful for developers if any want to expand on this project. The testerApplication.html is strictly for testing purposes only, as it only contains a very basic version of AR.js. This can be used to test if the device any developer is aiming for is suitable for the task, as it is always run on the GitHub pages website for the repository found here. The Documents folder is where all the documents required for the University module this project was made for are found. This won't be very useful to future developers.

The last item found in the home page of the repo is the sourceCode folder, which is the most important folder for this project. It contains all the source code for the whole AR application. Each source code file, whether it be HTML, Python or JavaScript, will contain comments to guide the developer, however this guide will still explain each file in laymans terms, ish. If you enter the folder, you will see 3 items: the static folder, the templates folder, and main.py. Now some developers may recognise this as the layout required for the Python web framework library Flask to work, however I will explain that later. The main.py file is what the Raspberry Pi runs to boot the web framework. It only contains 32 lines of code, but can be confusing to the unexperienced. More on the Flask library can be found in its documentation here.

The 2 other items in the sourceCode folder are used by the main.py file in order for Flask to work. The names of each folder are telling Flask what each one is used for. Static is used to store all the JavaScript and CSS files the HTML files need, whereas the Templates folder is used to store all the HTML files. If you look back to the main.py file, you can see each page is rendered using a function called render_template. This function only means that it's going to load the HTML file given in the parameters of the function from the Templates folder, hench the function name render_template.

Next up is the static folder. If you enter the folder you will be able to see all the JavaScript and the CSS files that the AR application uses, minus the external ones. To start this folder, I will explain Object.js first. Object.js is where the base object's details can be instantiated from. This allows me to just call on this class and use the costructor to create a new IRL object details object when the user wants to create a new one. The code also contains the savings and loading of the objects from session storage. Session storage is interal storage used by your browser to save information temporarily, and is lost when the session, a.k.a the tab you're on, is closed. This may be used to store how many times you clicked a button on a website etc. There is also local storage that still stores it in the browser, but the information is not lost when the session is closed, only when the cache is cleared in the browser. I used session storage only because the application is still in development and means I can reset all of the objects easily by just closing the tab. Next up is the OIP.js file. This file holds some essential functions including changing the url of the users browser when the user presses the specific buttons, e.g. 'Close' from the forms will take the user back to the main AR page. This will trigger the main.py to kick in when it detects the url change and will use it's render_template function to load up the new page. It also contains the methods to retrieve all the data of the objects to display on the AR page. The function returnObjectInfouses the other functions searchForObj and rtnObjjCharacs to return an Object's information. These functions use the Object's marker numbers as unique ID, as each marker is unique to each object, to find the object's and return their information.

The next JavaScript files in the static folder are used to give the forms functionality. So CreateNewObject.js is used with createNewObject.html, and ChangeObjectDetails.js is used with changeObjectDetails.html, etc. These JavaScript files extract the inputs from their corresponding HTML form to do their job, so CreateNewObject.js uses the input from createNewObject.html to create a new object with the name that the user input, ChangeObjectDetails.js uses the input from changeObjectDetails.html to change the details of the given object by using it's object marker number to find it in the array allObjs and change its details, and DeleteObject.js uses the input from deleteObject.html to delete the given object by, again, using it's object marker number to find it in allObjs and delete it. The last file in the static folder is the main.css. This is the css file for all the HTML files in the Templates folder. It sets the style of all the buttons on the AR page, all the forms and all of the input boxes and buttons in those forms. Each style is commented so if needed, a developer can understand which one is used when and why.

Now to move onto the Templates folder. As I mentioned earlier, the Templates folder contains all the HTML files needed for the AR application to run. These include: index.html, createNewObject.html, changeObjectDetails.html and deleteObject.html. The main file is index.html. This file contains the whole of the AR page, from importing AR.js and Aframe to using the JavaScript files to display the Object's OIPs (Object Information Panels). Although this file looks big for a small project like mine, each marker has been hardcoded to what object it refers to, and there are 10 markers. In an indeal situation where I had more time to develop this application further, I would figure out a way for this file to be of variable size, with markers only being in the code if they have an object assigned to them. Sadly, this is not the case so I had to hardcode each marker with what index in allObjs it should refer to when pulling its object's details. The next files in the Templates folder are the HTML files for the forms I spoke about earlier on. They simply create a HTML form with text boxes for the user to input the corresponding information, so if the HTML file is the createNewObject.html, then the form will contain a single text box asking for the user to enter the object's name, or if the HTML file is the changeObjectDetails.html, then the form will have 3 text boxes for the Object they want to change's marker number, the new name and the new object characteristics. The HTML files also contain the buttons for the user to either save the information they entered into the text boxes, or to close the form and return to the AR page.

Possible Future ideas

  • Implement a feature so that the application dos not contain a fixed number of objects, and that the markers are not hardcoded to their objects.
  • Implement a feature to allow the user to change the colour of the OIPs and text.
  • Implement a feature that displays all the objects from the menu.   

I hope that those ideas inspire you to use my project and produce new projects for yourself and others. Thank you for reading and be sure to check out my repository on GitHub!
